# TimeOps MVP 设计文档

- 日期：2026-05-17
- 状态：已在对话中确认，现整理为书面设计供审阅
- 范围：本文档仅描述 MVP 设计，不包含实现代码

## 1. 产品定位

TimeOps 是一个内部运维平台，用于在客户购买产品后，统一管理标准化部署、后续更新以及日常维护操作。

本期 MVP 明确服务于以下场景：

- 仅供内部人员使用
- 目标服务器为 Linux 主机
- 连接方式为 SSH 用户名密码
- 主要部署形态为 Docker Compose 应用
- 交付来源同时支持 Git 和上传发布包
- 部署后的核心运维能力优先支持应用更新
- 允许执行高风险临时命令，但必须全量审计
- 需要多内部账号与基于角色的权限控制

这个产品不是通用 SSH 面板，也不是完整 DevOps 平台。它的核心定位是一个围绕“产品模板、客户实例、远程执行、审计追踪”建立起来的标准化交付控制台。

## 2. MVP 目标

MVP 需要把以下流程做成可靠、可重复、可追踪的标准操作：

1. 录入客户及其服务器
2. 定义可复用的产品部署模板
3. 基于模板创建客户部署实例
4. 向客户服务器执行首次部署
5. 后续通过 Git 或发布包执行应用更新
6. 对目标实例执行授权内的运维动作
7. 对所有敏感操作建立完整审计链路

## 3. 明确不做的范围

本期 MVP 不包含以下内容：

- 客户侧门户
- 正式的多服务器环境编排
- 自动监控与告警
- 完整回滚体系
- 浏览器交互式终端
- CI/CD 流水线编排
- 灰度发布、金丝雀发布或分批发布

这些能力后续都可以扩展，但不应进入当前 MVP 范围。

## 4. 推荐的产品方向

在设计时评估了三条路线：

1. 轻量远程运维台
2. 标准化交付平台
3. 完整 DevOps / PaaS 平台

推荐采用第 2 条路线：标准化交付平台。

原因如下：

- 你的业务问题不只是远程执行命令
- 同一类产品会重复卖给不同客户，但每个客户会带有差异配置
- 真正有价值的是“交付、更新、审计”这一整条链路
- 用“模板 + 实例”的产品模型，比单纯按机器管理更贴合实际业务

## 5. 总体架构

MVP 建议采用“模块化单体 + 异步任务执行”的架构，而不是一开始拆微服务。

### 5.1 主要组成

- 管理后台 Web 应用
- 后端控制面 API
- 异步任务执行器
- 关系型数据库
- 本地发布包存储

### 5.2 为什么采用这种形态

这个平台前期的复杂度主要集中在：

- 远程任务执行
- 凭据安全
- 权限校验
- 审计记录

这些问题并不要求一开始拆成多个服务。模块化单体更容易实现、更容易维护，也更适合第一期快速打稳业务边界。

### 5.3 架构职责概述

- Web 后台：内部运维操作界面
- 后端 API：承载业务规则、RBAC、凭据处理、审计落库
- 任务执行器：通过 SSH 执行远程动作，并将结构化结果回传后端
- 数据库：保存元数据、配置、任务、用户、角色、审计记录
- 发布包存储：保存上传的制品文件，并与发布记录关联

## 6. 核心领域模型

MVP 建议建立以下核心对象。

### 6.1 客户 Customer

表示一个购买你产品的客户组织。

主要字段：

- 名称
- 联系人
- 联系电话或邮箱
- 合同开始时间
- 合同到期时间
- 备注

### 6.2 服务器 Server

表示客户提供的一台 Linux 服务器。

主要字段：

- 客户 ID
- 主机地址或 IP
- SSH 端口
- SSH 用户名
- 加密后的 SSH 密码
- 操作系统标签
- 机器标签
- 最近一次连通状态
- 备注

规则：

- 一个客户可以有多台服务器
- 密码必须加密存储，页面中不回显明文

### 6.3 产品模板 ProductTemplate

表示一类标准产品的交付定义。

主要字段：

- 模板名称
- 产品编码
- 支持的发布来源类型
- 默认工作目录
- 默认环境变量
- 默认配置结构
- 描述

作用：

- 描述这类产品通常如何部署和更新
- 将产品默认规则与客户差异配置分离

### 6.4 模板动作 TemplateAction

表示某个产品模板下可执行的标准动作。

首期动作类型建议包括：

- 首次部署
- 应用更新
- 服务重启
- 状态检查

执行模式分为：

- 脚本模式
- 步骤模式

MVP 规则：

- 实现可以先以脚本模式为主
- 但数据模型必须预留步骤模式，以便后续升级为结构化编排

### 6.5 部署实例 DeploymentInstance

表示某个客户在某台服务器上的一个实际部署对象。

主要字段：

- 客户 ID
- 模板 ID
- 主服务器 ID
- 实例名称
- 环境标签
- 当前运行版本 ID
- 状态
- 备注

示例：

- A 客户 / 官网系统 / 生产环境
- B 客户 / 管理后台 / 生产环境

这是整个平台最核心的运维对象。

### 6.6 实例配置 InstanceConfig

用于保存客户级覆盖配置。

典型内容：

- 域名
- 安装目录
- 端口
- 授权码
- 主题
- 功能开关
- 第三方集成参数

规则：

- 实际运行配置由“模板默认值 + 实例覆盖值”合并得到

### 6.7 发布版本 Release

表示一个可被部署的版本。

支持的来源类型：

- Git 来源
- 上传发布包

典型字段：

- 模板 ID
- 版本号或版本标签
- 来源类型
- 仓库地址或发布包文件引用
- Git 分支、标签或 commit
- 版本说明
- 创建人
- 创建时间

### 6.8 运维任务 OperationTask

表示一次实际执行的操作。

任务类型建议包括：

- 部署
- 更新
- 重启
- 状态检查
- 临时命令执行

主要字段：

- 任务编号
- 发起人用户 ID
- 目标部署实例 ID 或服务器 ID
- 动作引用
- 版本引用（如适用）
- 命令输入（如适用）
- 状态
- 开始时间
- 结束时间
- 退出码
- 标准输出日志
- 标准错误日志

这个对象同时是任务中心和审计体系的锚点。

## 7. 功能模块划分

MVP 后台建议包含以下模块。

### 7.1 仪表盘

用途：

- 展示最近任务数、成功率、失败任务、待处理问题
- 提供部署、更新、任务查看等快捷入口

### 7.2 客户管理

用途：

- 维护客户资料
- 查看该客户名下服务器、部署实例、最近操作记录

### 7.3 服务器管理

用途：

- 录入 SSH 目标
- 检测连通性
- 管理机器标签和基础信息
- 展示最近一次连通结果和最近任务状态

约束：

- 密码展示必须脱敏
- 修改密码应作为独立审计动作处理

### 7.4 产品模板管理

用途：

- 维护部署默认规则
- 定义标准动作与脚本
- 管理配置结构与支持的发布来源

### 7.5 部署实例管理

用途：

- 将客户、模板、服务器、差异配置绑定成一个可运维对象
- 从实例维度发起部署、更新、重启、状态检查、临时命令

### 7.6 发布版本管理

用途：

- 维护 Git 型发布版本
- 上传并管理发布包
- 将版本与产品模板关联

### 7.7 任务中心

用途：

- 统一查看所有运维任务的队列与历史
- 按任务类型、状态、用户、客户、服务器、时间过滤
- 查看执行日志、耗时、结果

### 7.8 审计日志

用途：

- 记录所有敏感操作
- 支持追踪登录、凭据修改、部署、更新、删除、命令执行等行为

### 7.9 用户与角色

用途：

- 管理内部账号
- 分配角色
- 控制操作权限

## 8. 导航结构建议

后台一级导航建议如下：

- 仪表盘
- 客户
- 服务器
- 产品模板
- 部署实例
- 发布版本
- 任务中心
- 审计日志
- 用户与角色

这个结构符合日常使用顺序：先建模板，再录客户和服务器，再创建部署实例，然后围绕版本和任务持续交付与维护。

## 9. 关键业务流程

### 9.1 首次部署流程

1. 创建客户
2. 录入服务器
3. 基于产品模板创建部署实例
4. 选择发布来源：Git 或发布包
5. 合并模板默认配置与实例覆盖配置
6. 通过 SSH 连接目标服务器
7. 拉取或上传部署资产
8. 生成运行时配置文件
9. 执行部署动作
10. 保存任务输出与最终结果

### 9.2 应用更新流程

1. 打开某个部署实例
2. 选择目标发布版本
3. 执行模板定义的更新动作
4. 拉取新代码或上传新包
5. 按需要保留或更新配置
6. 重启 Docker Compose 服务
7. 成功后更新实例当前运行版本
8. 保存任务日志与结果

### 9.3 临时命令执行流程

1. 选择目标服务器或部署实例
2. 输入命令
3. 校验权限
4. 要求风险确认
5. 通过 SSH 执行
6. 保存命令内容、输出、退出码、发起人、时间等信息

### 9.4 审计关联流程

每一个重要操作都必须能完整追踪到：

- 谁发起的
- 影响了哪个目标
- 执行了什么动作
- 使用了哪个模板、版本或命令
- 开始和结束时间
- 执行结果
- 输出日志

审计记录需要能够反查到具体任务详情。

## 10. 安全要求

由于平台会保存客户服务器凭据，并具备远程执行能力，安全设计必须是首要能力，而不是后补能力。

### 10.1 凭据处理

- SSH 密码必须加密存储
- 页面中不得显示明文密码
- 密码变更必须单独进入审计
- 只有在执行任务时才在后端内存中短暂解密使用

### 10.2 权限控制

MVP 至少应支持以下角色：

- 超级管理员
- 实施 / 交付
- 运维
- 审计 / 只读

权限粒度建议做到：

- 模块级访问
- 动作级权限

特别要求：

- “临时命令执行”必须是独立权限点

### 10.3 高风险操作控制

以下操作应要求二次确认：

- 删除服务器
- 修改凭据
- 手工重试失败任务
- 执行临时命令

### 10.4 输出脱敏

任务日志和审计日志应对以下内容做脱敏处理：

- 密码
- Token
- API Key
- 环境变量中的敏感信息

否则日志本身会成为泄漏面。

## 11. 审计要求

审计系统是平台核心能力之一，不是附属功能。

至少需要记录：

- 谁执行了操作
- 影响了哪个目标
- 执行了什么动作
- 使用了哪个模板、版本或命令
- 何时开始
- 何时结束
- 是否成功
- 对应输出和退出码

其中“临时命令执行”必须支持单独筛选和高亮展示，因为其风险更高。

## 12. 技术落地建议

考虑到该平台偏企业后台、权限体系复杂、审计要求高、需要长期维护，技术选型应优先考虑结构清晰和后续稳定演进，而不是只追求短期开发速度。

### 12.1 推荐技术栈

- 前端：React + Ant Design
- 后端：Spring Boot
- 数据库：MySQL 或 PostgreSQL
- 任务执行：后端统一管理的异步任务队列
- 远程执行：后端使用成熟 SSH 库
- 发布包存储：MVP 先采用本地受管存储

### 12.2 推荐 Spring Boot 的原因

- 更适合企业后台系统
- 在 RBAC、审计、事务、模块化业务边界上更成熟
- 对复杂权限和任务执行链路的长期维护更友好
- 后续如果任务执行器需要独立扩展，也容易演进

### 12.3 后端模块边界建议

- 认证与用户
- 角色与权限
- 客户
- 服务器
- 产品模板
- 部署实例
- 发布版本
- 运维任务
- 审计
- 凭据加密

这些模块可以先放在一个后端服务内，但内部边界要清晰。

## 13. 数据与执行设计原则

实现阶段应保持以下原则稳定不变：

1. 产品定义必须与客户差异配置分离
2. 运维动作应围绕部署实例展开，而不是裸服务器
3. 所有远程执行必须纳入任务模型
4. 敏感操作必须天然可审计
5. 数据模型要允许未来从单机扩展到多机，而不推翻核心实体

## 14. 面向未来的扩展点

当前模型应兼容后续扩展：

- 服务器组或多节点环境
- 基于步骤的流程编排
- 回滚快照
- 监控与告警集成
- 客户状态门户
- 对象存储保存发布包

## 15. 建议的实施顺序

进入实现阶段后，建议按以下顺序推进：

1. 认证、用户、角色、权限框架
2. 客户与服务器管理
3. 产品模板与模板动作
4. 部署实例与实例配置
5. 发布版本
6. 任务执行引擎与 SSH 执行能力
7. 部署与更新流程
8. 审计视图与报表

这个顺序可以减少返工，因为任务执行和审计依赖前面的核心对象先稳定下来。

## 16. 最终 MVP 总结

本次确认的 MVP 范围包括：

- 客户管理
- 服务器管理
- 产品模板管理
- 部署实例管理
- 发布版本管理
- 首次部署
- 应用更新
- 临时命令执行
- 任务中心
- 审计日志
- 用户与角色

本期 MVP 应实现为一个面向企业后台场景的运维交付平台，后端以 Spring Boot 为核心，前端采用 React + Ant Design，围绕“产品模板、客户实例、异步 SSH 执行、严格审计”建立稳定的一期能力。
